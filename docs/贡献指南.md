# 贡献指南（团队协作规范）

> 目标：多人同时开发 15 个功能模块时互不踩脚。规则不多，但请全组遵守。
> 配套的课程文档（配置管理计划书、需求规约等）见 `工程文档-指导版/`。

## 1. 分工建议

按**功能模块**分工（而不是按前端/后端分工）：一人认领一个模块，
从数据库表到后端到网站端到小程序端整条链路自己走通 —— news 模块就是模板，
步骤见 [docs/新增功能指南.md](./新增功能指南.md)。这样每个人都能练到全栈，答辩时也说得清楚自己做了什么。

## 2. Git 工作流

### 分支模型

```
main                 ← 随时可运行的稳定分支，只接受 PR 合入，不直接 push
└── feature/xxx      ← 每个功能一个分支，如 feature/notice、feature/login
└── fix/xxx          ← 修 bug 分支，如 fix/news-page-size
```

### 日常操作

```bash
git checkout main && git pull            # 1. 先同步最新 main
git checkout -b feature/notice           # 2. 从 main 拉出功能分支
# ... 开发、提交（提交粒度小一点，一个提交做一件事）...
git push -u origin feature/notice        # 3. 推到远端
# 4. 在 GitHub 上发起 Pull Request -> 找一名同学 review -> 合入 main
```

### 提交信息规范

格式：`类型: 中文描述`（描述写清"做了什么"，不要只写"修改"、"更新"）

| 类型 | 用途 | 示例 |
| --- | --- | --- |
| feat | 新功能 | `feat: 公告模块后端接口` |
| fix | 修 bug | `fix: 新闻列表分页参数从 0 开始导致查不到数据` |
| docs | 只改文档 | `docs: 补充小程序真机调试说明` |
| refactor | 重构（不改行为） | `refactor: 抽取公告查询公共条件` |
| chore | 构建/配置等杂项 | `chore: 更新 .gitignore` |

## 3. 代码规范

### 后端（分层职责，越界的 PR 会被打回）

| 层 | 只允许做 | 不允许做 |
| --- | --- | --- |
| api（Controller） | 收参数、调 AppService、包 `Result` | 写业务逻辑、直接碰 Mapper |
| application | 用例编排、DTO 组装、事务边界 | 写 SQL、绕过领域实体直接改状态 |
| domain | 业务规则（实体方法）、仓储接口 | import 任何 Spring/MyBatis 的类 |
| infrastructure | PO、Mapper、仓储实现、PO↔实体转换 | 对外暴露 PO（出口必须是领域实体） |

- 统一返回 `Result` / 分页统一 `PageResult`，不要自己发明返回格式；
- 可预期的业务错误抛 `BusinessException`，全局异常处理器会兜住；
- 类和公开方法写中文 Javadoc，说明"它是干嘛的、新增功能时怎么参照"。

### 前端（web / miniapp）

- 接口调用**只**写在 `api/` 目录（一个功能一个文件），页面里不直接 axios / uni.request；
- 网站端组件名用大驼峰多单词（`NewsListView.vue`），路由 path 用小写中划线；
- 与后端约定的字符串常量（如新闻栏目）定义在 api 文件里 export，不要在页面里写死多份；
- 别改 `utils/request.*` 的拆包约定；新需求先在群里商量。

### 数据库

- 建表脚本进 `docs/sql/`，文件名 `编号_功能.sql` 递增，**不改别人已合入的脚本**；
- 表名 `t_功能名`，列名下划线风格，标配 `create_time`/`update_time`/`deleted` 三列（参照 `t_news`）。

## 4. 提 PR 前自检

```bash
# 后端能编译
cd backend && mvn -q compile
# 网站端类型检查 + 打包通过
cd web && npm run build
# 小程序端在微信开发者工具里手动跑一遍主流程
```

外加 [docs/新增功能指南.md](./新增功能指南.md) 末尾的「联调自测清单」。

## 5. 遇到问题

1. 先看 news 模块对应位置的写法和注释（每个文件头都写了"新增功能时怎么抄"）；
2. 再看 [docs/新增功能指南.md](./新增功能指南.md) 的「常见坑」表；
3. 还不行就在群里贴 **报错信息 + 复现步骤**（不要只发一句"我这里报错了"）。
